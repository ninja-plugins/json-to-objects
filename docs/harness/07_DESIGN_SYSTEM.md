# 07. 디자인 시스템 하네스

> 테마 이름, 토큰 접두사, 브랜드 색상, 글꼴 묶음, 호환 별칭은 프로젝트 프로필 값이다. 이 문서는 역할 기반 범용 기준만 둔다.

이 문서는 프론트엔드 화면 영역의 시각 품질 기준을 정의한다. 주요 운영 프론트엔드, 보조 앱, 공개/인증 화면, 대시보드, 공유 UI 컴포넌트에 적용한다.

## 빠른 체크리스트

- 프로젝트가 소유한 스타일만 사용한다. 명시 승인 없이 외부 스타일/UI 프레임워크, 유틸리티 CSS 프레임워크, 복사한 디자인 키트, CDN 스타일시트를 추가하지 않는다.
- 원시 색상, 글꼴 체계, 반경, 그림자, 간격은 넓게 쓰는 컴포넌트에 적용하기 전에 토큰 또는 로컬 스타일 변수로 정의한다.
- 주요 행동 색상, 링크 색상, 의미 색상, 포커스 링, 파괴적 행동 색상은 서로 다른 역할이다.
- 운영 화면은 밀도 있고, 조용하고, 훑기 쉽고, 업무에 집중되어야 한다.
- 표현 중심 구도는 인증, 온보딩, 빈 상태, 데모, 공개 화면, 설명형 화면에만 사용한다.
- 사용자에게 보이는 문구는 프로젝트의 i18n/content 시스템을 따른다.
- 상태 표시는 색상만으로 의미를 전달하지 않는다.
- 모바일/터치 맥락은 프로젝트 프로필이 더 엄격한 기준을 두지 않는 한 최소 44px x 44px 터치 영역을 유지한다.
- 포커스 상태, 움직임 줄이기, 다크/고대비 변형, 안전 영역, 텍스트 비겹침을 확인한다.
- 동작 변경은 `05_TESTING.md`와 `09_EVIDENCE_GATE.md`를 따른다. 순수 시각 변경은 문서화된 diff/rg/browser 검증으로 대체할 수 있다.

## 필수 읽기 문서

- `AGENTS.md` 또는 `CLAUDE.md`
- 주요 프론트엔드 작업: `docs/harness/02_PRIMARY_FRONTEND.md`
- 보조 앱 작업: `docs/harness/03_SECONDARY_APP.md`
- `docs/harness/context/frontend/README.md`
- `docs/harness/profiles/design-system-profile.md`
- `.agents/skills/design-system/SKILL.md`

## 적용 범위

아래 작업에 이 문서를 적용한다.

- 새 화면, 공유 컴포넌트, 시각 리팩토링
- 버튼, 배지, 카드, 모달, 목록, 표, 폼, 내비게이션, 검색, 필터, 페이지네이션, 도구막대
- 글꼴 체계, 간격, 반경, 그림자, 색상 역할, 반응형 동작
- 디자인 리뷰, 반응형 리뷰, UX 리뷰, 데이터 시각화 리뷰, 콘텐츠/i18n 리뷰
- 인증, 온보딩, 빈 상태, 공개 화면, 데모, 문서 인접 UI 영역

좁은 버그 수정은 불필요한 변경을 피하기 위해 기존 스타일을 보존할 수 있다. 새 UI는 이 기준을 따른다.

## CSS 소유권

- UI 스타일은 프로젝트가 소유한 CSS/SCSS, 로컬 스타일 모듈, 또는 프로젝트의 기존 스타일 시스템에서 나온다.
- 레이아웃이나 시각 마감을 위해 서드파티 테마 CSS나 생성된 프레임워크 클래스를 가져오지 않는다.
- 아이콘 라이브러리와 도메인 라이브러리는 레이아웃 목적이 아닌 용도로만 허용한다. 컴포넌트 형태, 간격, 색상, 타입, 반응형 동작, 상호작용 스타일은 앱이 직접 소유한다.
- 의존성이 필수 CSS를 제공한다면 벤더 통합으로 명시 검토하고, UI 기반으로 삼지 않는다.

## 토큰 정책

- 핵심 문서는 프로젝트 토큰 이름이 아니라 의미 역할을 사용한다.
- 디자인 시스템 프로필은 이 역할을 실제 토큰 이름, 값, 별칭, 마이그레이션 규칙에 매핑한다.
- 컴포넌트는 원시 값을 인라인으로 쓰지 말고 로컬 디자인 시스템을 통해 역할 토큰을 사용한다.
- 레거시 코드 호환을 위한 별칭은 둘 수 있지만, 새 컴포넌트는 의미 역할 이름을 우선한다.

### 필수 역할 그룹

| 역할 그룹 | 예시 | 사용처 |
|---|---|---|
| 브랜드 | primary, primary-hover, primary-pressed, primary-subtle | 주요 행동, 선택 상태, 브랜드 강조 |
| 링크 | link, link-hover, link-visited | 인라인 내비게이션과 텍스트 링크 전용 |
| 표면 | canvas, surface, surface-raised, surface-muted | 페이지 배경, 카드, 패널, 선택 영역 |
| 테두리 | hairline, border-muted, border-strong, focus-ring | 구분선, 입력, 포커스 윤곽선 |
| 텍스트 | text-primary, text-secondary, text-muted, text-inverse | 전경 계층 |
| 의미 | success, warning, danger, info | 상태와 검증. 항상 텍스트/아이콘과 함께 사용 |
| 강조 | accent-1 through accent-n | 비의미 차트/분류/장식 강조 |
| 오버레이 | scrim, popover-shadow, modal-shadow | 메뉴, 모달, 떠 있는 패널 |

## 타이포그래피

- 글꼴 묶음과 정확한 척도는 프로젝트 프로필 값이다.
- 고정/rem 척도, 반응형 분기점, 컨테이너 제약을 사용한다. 뷰포트 너비에 직접 글꼴 크기를 연결하지 않는다.
- 자간은 프로젝트에서 검증한 토큰이 없으면 `0`을 기본값으로 둔다.
- 밀도 높은 운영 화면은 압축된 본문/표 척도를 사용할 수 있고, 표현형 화면은 더 큰 표시 척도를 쓸 수 있다.
- 버튼, 배지, 표 셀, 카드, 도구막대 컨트롤은 가장 긴 예상 현지화 문구를 넣어도 겹치지 않아야 한다.
- 글꼴 토큰은 필요에 따라 표시, 제목, 본문, 보조문구, 라벨, 버튼, 고정폭 역할을 포함한다.

## 레이아웃과 밀도

- 일관된 간격 척도를 사용한다. 프로필이 별도 척도를 정하지 않았다면 4px 기준과 8px 주요 증가 폭을 기본값으로 본다.
- 운영 화면은 scan, comparison, repeated action, keyboard/mouse efficiency, predictable navigation을 우선한다.
- 표현형 화면은 더 큰 간격과 강한 계층을 사용할 수 있지만, 필요할 때 실제 제품/데이터/작업 흐름/대상이 드러나야 한다.
- 그리드, 보드, 카운터, 표, 도구막대, 타일처럼 고정 형식 UI는 호버/로딩/라벨 변화로 레이아웃이 밀리지 않게 안정된 크기를 둔다.
- 카드 안에 카드를 넣지 않는다. 카드는 반복 항목, 모달, 실제 도구 프레임에 사용하고, 페이지 섹션은 전체 너비 섹션 또는 프레임 없는 레이아웃으로 둔다.
- 프로젝트 프로필이 명시하지 않는 한 순수 장식 그라데이션, 덩어리형 배경 장식, 단일 색조 팔레트를 피한다.

## 형태와 깊이

- 버튼, 입력, 검색 필드는 보통 중간 반경의 직사각형 컨트롤로 둔다.
- 카드와 패널은 절제된 반경과 얇은 테두리를 기본으로 한다.
- 알약형 형태는 배지, 탭, 세그먼트 컨트롤, 상태 칩, 원형/아이콘 컨트롤에만 사용한다.
- 운영 카드와 표는 대체로 평평하게 유지한다. 강한 그림자는 모달, 드롭다운, 실제 산출물 미리보기에 남긴다.
- 포커스 윤곽선은 보여야 하며 토큰화되어야 한다.

## 컴포넌트 규칙

### 버튼

- 하나의 화면 영역 또는 컴포넌트 그룹에는 지배적인 주요 행동을 하나만 둔다.
- 주요, 보조, 고스트, 파괴적, 링크, 아이콘 전용, 어두운 배경용 변형은 기본, 호버, 포커스, 활성, 로딩, 비활성 상태를 가진다.
- 비활성 상태는 불투명도에만 의존하지 않는다.
- 아이콘 전용 버튼에는 접근 가능한 라벨과 호버/포커스 단서가 필요하다.

### 폼

- 입력은 라벨, 도움/오류 문구, 포커스 상태, 비활성/읽기 전용 상태, 검증 상태를 가진다.
- 검증은 텍스트를 포함해야 하며 색상만으로 의미를 전달하지 않는다.
- 필수/선택 의미가 완료에 영향을 주면 명시한다.

### 카드와 목록

- 카드는 독립 대상, 선택지, 요약, 프레임이 필요한 도구를 나타낼 때 사용한다.
- 목록/표는 행 높이, 구분선, 선택 상태, 호버/포커스 상태, 빈 상태, 로딩 상태가 안정적이어야 한다.
- 반복 대상 카드마다 새로운 시각 문법을 만들지 않는다.

### 내비게이션

- 내비게이션은 대상 앱의 기존 셸을 따른다.
- 활성, 호버, 포커스, 접힘/모바일, 넘침 동작이 명확해야 한다.
- 모바일/터치 내비게이션은 터치 영역과 안전 영역 제약을 지킨다.

### 상태와 Badge

- 상태 표시는 색상과 텍스트, 아이콘, 형태 중 하나 이상을 함께 사용한다.
- 의미 상태 토큰은 브랜드/강조 토큰과 분리한다.
- 배지 라벨은 현지화 문구를 넣어도 작은 크기에서 읽을 수 있어야 한다.

### 데이터 시각화

- 의미 팔레트와 분류 팔레트를 의도적으로 구분한다.
- 차트에는 의미 있는 라벨, 범례, 툴팁, 빈/오류 상태, 접근 가능한 요약이 필요하다.
- 중요한 시리즈나 상태를 색상만으로 구분하지 않는다.

## 반응형 동작

| 범위 | 기본 동작 |
|---|---|
| 작은 모바일 | 단일 열, 압축된 컨트롤, 가로 넘침 없음 |
| 큰 모바일 | 라벨과 컨트롤이 맞을 때만 2열 허용 |
| 태블릿 | 필요 시 2열 레이아웃과 압축된 내비게이션 |
| 데스크톱 | 밀도 높은 운영 레이아웃, 표, 사이드 패널, 다열 요약 |
| 넓은 데스크톱 | 단순히 줄 길이를 늘리지 말고 비교, 맥락, 미리보기에 추가 공간 사용 |

- 지원 뷰포트에서 텍스트가 다른 요소와 겹치면 안 된다.
- 내비게이션과 도구막대는 컨트롤이 충돌하기 전에 넘침 동작을 가져야 한다.
- 모달, 드로어, 떠 있는 패널은 모바일 대안 또는 반응형 크기 조정을 가진다.
- 의미 있는 UI 변경은 가장 작은 지원 너비와 대표 데스크톱 너비에서 브라우저 검증을 수행한다.

## 해야 할 것

- 의미 역할 토큰과 프로필 정의 매핑을 사용한다.
- 운영 작업 흐름을 밀도 있고 예측 가능하며 훑기 쉽게 유지한다.
- 사용자에게 보이는 문구는 i18n/content 키를 사용한다.
- 로딩, 빈 상태, 오류, 재시도, 비활성, 포커스, 선택 상태를 제공한다.
- 새 변형을 만들기 전에 기존 로컬 컴포넌트를 확인한다.
- 대상 뷰포트 범위에서 텍스트 맞춤과 상호작용 상태를 검증한다.

## 하지 말 것

- 핵심 하네스 문서에 프로젝트별 색상, 토큰 접두사, 테마 이름, 패키지 이름, 행위자 라벨을 하드코딩하지 않는다.
- 승인 없는 외부 UI/스타일 프레임워크를 추가하지 않는다.
- 일반 버튼을 기본적으로 알약형으로 만들지 않는다.
- 상태나 검증을 색상만으로 표현하지 않는다.
- 운영 작업 흐름 화면을 마케팅형 첫 화면처럼 만들지 않는다.
- 런타임, 밀도, 권한 모델이 다른 화면 영역의 스타일을 직접 가져오지 않는다.
- 고정 데모 데이터나 미검증 뷰포트 가정으로 레이아웃 불안정을 숨기지 않는다.

## 검증

- 실제 명령은 `docs/harness/context/BASELINE.md` 또는 프로젝트 프로필에서 가져온다.
- 공유 스타일/토큰 변경은 관련 프론트엔드 테스트/빌드와 로컬 스타일 빌드 명령을 실행한다.
- 의미 있는 시각 변경은 대표 데스크톱/모바일 뷰포트에서 브라우저 검증이 필요하다.
- 런타임별 UI 변경은 대상 런타임 검증 또는 명시적 예외가 필요하다.
- 순수 시각 예외는 스크린샷/브라우저 메모, `rg` 확인, diff 리뷰로 활성 계획에 기록한다.
