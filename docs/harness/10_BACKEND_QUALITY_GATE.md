# 10. 백엔드 구조 품질 게이트

이 문서는 백엔드 변경에서 **DDD, 트랜잭션 관리, OOP, SOLID, Clean Code**를 명시적으로 점검하기 위한 기준이다. `01_BACKEND.md`가 운영 기준이라면, 이 문서는 구조 품질 리뷰 기준이다.

## 적용 대상

아래 변경에는 이 게이트를 적용한다.

- 도메인, 애플리케이션 서비스, 리포지토리, 영속성 변경
- API 동작, 권한, 상태 전이, 일관성이 중요한 로직 변경
- 트랜잭션 경계, 잠금, 멱등성, 재시도, 콜백, 이벤트, outbox 변경
- 큰 리팩토링, 책임 분리, 패키지 이동, 공유 추상화 작업

오타, 문서, 순수 스타일 변경처럼 이 게이트가 유용하지 않으면 활성 계획에 예외 사유를 남긴다.

## 1. DDD 체크

- [ ] 도메인 언어가 코드 이름에 반영되어 있고, 핵심 개념이 기술 용어 뒤에 숨지 않는다.
- [ ] 도메인/애플리케이션/표현/인프라 경계가 유지된다.
- [ ] 표현 계층 코드는 요청/응답 매핑과 검증만 담당한다.
- [ ] 애플리케이션 서비스는 유스케이스 조율과 트랜잭션 경계를 담당하고 도메인 규칙을 중복 구현하지 않는다.
- [ ] 도메인 규칙과 불변 조건은 가능한 한 도메인 모델, 도메인 서비스, 정책, 값 객체 안에 둔다.
- [ ] 엔티티, 값 객체, 애그리거트, 리포지토리, 도메인 서비스, 정책, 애플리케이션 서비스 역할이 무리하게 섞이지 않는다.
- [ ] ORM 기반 모델이라도 도메인 불변 조건이 중요하면 무분별한 setter, mutable collection 노출, 의미 없는 데이터 묶음을 피한다.
- [ ] 리포지토리는 영속성 추상화 역할을 하며 표현/DTO 계층이 영속성 세부사항을 알지 않는다.
- [ ] 도메인 변경으로 장기 사실이 바뀌면 관련 `docs/harness/context/backend/domains/*.md`를 갱신한다.

## 2. 트랜잭션 관리 체크

- [ ] 쓰기 유스케이스는 명확한 애플리케이션/유스케이스 경계에서 트랜잭션을 시작한다.
- [ ] 조회 유스케이스는 스택이 지원하면 읽기 전용/쿼리 경계를 사용한다.
- [ ] 레거시/프레임워크 제약이 문서화된 경우가 아니라면 표현 계층 코드가 트랜잭션 조율을 소유하지 않는다.
- [ ] 프레임워크 트랜잭션 동작이 private method, self call, 비동기 경계, proxy boundary, nested call에서 실제 적용되는지 확인한다.
- [ ] 외부 API 호출, 파일 I/O, 메시징, 이메일, 알림 등 부수효과는 트랜잭션 내부 또는 커밋 후 실행 결정을 명시한다.
- [ ] 부수효과가 중요하면 커밋 후 이벤트, outbox, 재시도, 보상 정책을 검토한다.
- [ ] 중복 제출, 콜백 재시도, 네트워크 재시도는 멱등성 키, unique constraint, 상태 전이 조건으로 방어한다.
- [ ] 동시성 위험은 optimistic lock, pessimistic lock, unique constraint, distributed lock, compare-and-swap, queueing 또는 다른 문서화된 전략으로 처리한다.
- [ ] lazy loading, N+1, flush timing, cascade behavior, propagation, rollback rule이 선택한 스택에서 의도와 맞는지 확인한다.
- [ ] 트랜잭션/멱등성/동시성 변경은 테스트 또는 재현 절차로 검증한다.

## 3. OOP 체크

- [ ] 객체가 스스로 보호할 수 있는 규칙은 데이터와 행위를 과도하게 분리하지 않는다.
- [ ] 서비스가 절차적 catch-all이 되지 않도록 도메인 객체, 정책, 전략 책임을 필요에 따라 나눈다.
- [ ] 객체 불변 조건은 생성자, 팩토리, 도메인 메서드, 상태 전이 메서드에서 보호한다.
- [ ] public setter, public mutable collection, map/string/object 남발을 피한다.
- [ ] 반복 조건문을 줄이고 정확성을 높일 수 있으면 Tell, Don't Ask를 우선한다.
- [ ] primitive obsession이 정확성이나 가독성에 영향을 주면 값 객체 승격을 검토한다.
- [ ] utility/static helper는 순수 함수 또는 경계 변환에 제한하고 도메인 정책을 숨기지 않는다.

## 4. SOLID 체크

### SRP: 단일 책임 원칙

- [ ] 클래스와 메서드가 명확한 변경 이유를 가진다.
- [ ] God service, 컨트롤러, 매퍼, 어댑터가 되지 않는다.

### OCP: 개방-폐쇄 원칙

- [ ] 정책/타입 변형을 추가할 때 같은 조건문을 계속 고쳐야 한다면 전략, 정책, enum behavior, 다형성을 검토한다.
- [ ] 추측성 추상화는 금지한다. 실제 변형 축에만 확장 지점을 둔다.

### LSP: 리스코프 치환 원칙

- [ ] 하위 타입과 구현은 부모 계약의 사전 조건, 사후 조건, 오류 의미를 지킨다.
- [ ] 호출자가 올바르게 동작하기 위해 구현별 분기를 해야 하는 구조를 피한다.

### ISP: 인터페이스 분리 원칙

- [ ] 인터페이스가 호출자 필요보다 넓지 않다.
- [ ] 명령/조회, 읽기/쓰기, 외부 어댑터 계약은 결합도를 줄일 때 분리한다.

### DIP: 의존성 역전 원칙

- [ ] 도메인/애플리케이션 코드가 인프라 세부사항에 직접 의존하지 않는다.
- [ ] 외부 시스템, 시계, UUID, 저장소, 결제, 알림 등은 포트/어댑터 또는 명확한 경계를 지난다.
- [ ] 테스트나 대체 구현을 어렵게 하는 직접 생성, static call, 전역 상태를 피한다.

## 5. 클린 코드 체크

- [ ] 메서드는 한 가지 일을 하며 중첩 깊이가 과하지 않다.
- [ ] 이름은 의도를 드러내고 의미 없는 `data`, `info`, `tmp`, 헷갈리는 약어를 피한다.
- [ ] 의미 있는 중복은 로컬하게 추출한다.
- [ ] 정책을 담은 magic number/string은 named constant, enum, 값 객체로 표현한다.
- [ ] 주석은 "왜"를 설명하고, 다음 줄을 그대로 풀어쓰지 않는다.
- [ ] dead code, unused import/variable/parameter, unreachable branch를 새로 만들지 않는다.
- [ ] guard clause와 early return이 가독성을 높이면 흐름을 평탄하게 만든다.
- [ ] 요청 범위 밖 정리는 같은 변경에 섞지 않고 후속 작업으로 기록한다.

## 6. 리뷰 판정 기준

아래 항목이 남아 있으면 `FAIL` 또는 `PASS_WITH_CONCERNS`로 판정한다.

- 표현 계층 코드에 핵심 비즈니스 규칙이나 트랜잭션 조율이 들어 있다.
- 트랜잭션 경계가 불명확하거나 런타임에서 적용되지 않을 가능성이 있다.
- 도메인 규칙이 DTO, 매퍼, 리포지토리 쿼리, UI 편의 로직에 흩어져 있다.
- 상태 전이, 중복 요청, 동시성 위험에 테스트와 데이터 제약이 없다.
- God service, public setter 남발, map/string 기반 도메인 처리가 불변 조건을 깨뜨릴 수 있다.
- SOLID 위반이 변경 비용, 테스트 난이도, 결함 위험을 의미 있게 키운다.

## 도메인 / 영속성 분리 체크

- [ ] 도메인 규칙이 리포지토리 쿼리, 매퍼, DTO, 표현 계층 코드에 숨어 있지 않다.
- [ ] 필요하면 `backend-domain-modeler`가 애그리거트, 불변 조건, 상태 전이, 도메인 서비스, 정책 책임을 검토했다.
- [ ] 필요하면 `backend-persistence-implementer`가 ORM 매핑, 쿼리, 잠금, 페이지네이션, 영속성 테스트를 검토했다.
- [ ] 애플리케이션 트랜잭션 경계가 도메인/영속성 책임과 모순되지 않는다.
- [ ] 영속성 모델과 도메인 모델을 합쳐 쓰더라도 ORM 편의 때문에 도메인 불변 조건이 약해지지 않는다.
- [ ] 리포지토리 인터페이스는 도메인 언어를 유지하고, 구현이 인프라 세부사항을 가진다.
- [ ] 병렬 작업을 했다면 VERIFY 전에 도메인/영속성 불일치를 확인했다.
