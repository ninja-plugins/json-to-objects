# 11. 병렬 에이전트 게이트

이 문서는 여러 에이전트가 병렬로 작업해도 안전한지 판단한다. 먼저 `13_AGENT_ORCHESTRATION.md`에서 단일/순차/분리 위임 여부를 결정하고, 병렬 후보일 때만 이 문서를 적용한다. 독립성이 명확하지 않으면 기본값은 순차 실행이다.

## 목적

- 독립적인 리뷰나 구현 작업을 병렬화해 속도를 높인다.
- 같은 파일, 트랜잭션 경계, API 계약, 스키마, 도메인 규칙, 권한 정책을 여러 에이전트가 동시에 건드려 충돌하는 상황을 막는다.
- 모든 결과는 최종 VERIFY 전에 한 명의 통합자가 수렴한다.

## 원칙

- 디스패치 전에 병렬 가능 여부를 확인한다.
- 구현 병렬화보다 리뷰 병렬화를 우선한다.
- 계약, 파일 소유권, 의존성이 불명확하면 순차 실행한다.
- RED -> GREEN -> REFACTOR -> VERIFY 순서는 유지한다.
- 한 명의 통합자가 수렴과 최종 검증을 소유한다.

## 병렬 실행 가능 조건

아래 조건을 모두 만족할 때만 병렬 작업을 고려한다.

- 요구사항과 API/DTO/스키마 계약이 충분히 고정되어 있다.
- 에이전트별 파일 소유 범위가 겹치지 않는다.
- 한 에이전트의 산출물이 다른 에이전트의 필수 입력이 아니다.
- 트랜잭션 경계, 권한 정책, 도메인 불변 조건, 리소스 범위 규칙이 충돌하지 않는다.
- 활성 계획에 단일 통합자가 지정되어 있다.
- 에이전트별 산출물, 검증 방법, 보고 형식이 활성 계획에 분리되어 있다.

## 병렬 실행 금지 조건

아래 중 하나라도 해당하면 순차 실행한다.

- 두 에이전트가 같은 파일, 함수, 컴포넌트, 애그리거트, 라우트, 스키마 객체를 수정한다.
- 마이그레이션/스키마/데이터 시드 순서가 얽혀 있다.
- 공유 DTO, API 계약, enum, 권한 정책, 리소스 범위 규칙이 고정되지 않았다.
- 같은 애그리거트, 트랜잭션 경계, 멱등성 키, 잠금 정책, 상태 전이를 함께 바꾼다.
- RED 증거 없이 여러 구현 에이전트가 먼저 시작해야 한다.
- 요구사항이 모호하거나 안전하게 나눌 기준이 없다.
- 전역 설정, 빌드 설정, 환경 파일, 보안 설정을 수정한다.

## 오케스트레이션과의 관계

- `SINGLE_AGENT`, `SINGLE_AGENT_WITH_REVIEW`, `SEQUENTIAL_LAYERED`는 병렬 게이트 없이도 선택할 수 있다.
- `PARALLEL_INVESTIGATION`, `PARALLEL_REVIEW`, `PARALLEL_IMPLEMENT`는 이 문서의 금지 조건을 통과해야 한다.
- 레이어별 에이전트가 존재한다는 이유만으로 병렬 실행하지 않는다.

## 권장 모드

| 모드 | 사용 조건 | 예시 |
|---|---|---|
| `SEQUENTIAL` | 의존성이나 충돌 위험이 의미 있게 있음 | 스키마를 먼저 고친 뒤 리포지토리/서비스/컨트롤러 순차 처리 |
| `PARALLEL_REVIEW` | 구현은 끝났고 독립 리뷰가 유용함 | 보안, 통합, 접근성, 품질 리뷰 |
| `PARALLEL_IMPLEMENT` | 계약이 고정됐고 파일 범위가 겹치지 않음 | 백엔드 제공자와 프론트엔드 소비자를 서로 다른 파일에서 구현 |
| `PARALLEL_INVESTIGATION` | 구현 전 조사를 안전하게 나눌 수 있음 | 백엔드 영향, 프론트엔드 영향, 테스트 영향 조사 |

## 절차

1. 활성 계획에 `Parallelization Check`를 추가한다.
2. `SEQUENTIAL`, `PARALLEL_REVIEW`, `PARALLEL_IMPLEMENT`, `PARALLEL_INVESTIGATION` 중 하나를 고른다.
3. 병렬 실행 시 각 에이전트의 범위, 수정 가능 파일, 읽기 전용 파일, 검증, 출력 형식을 적는다.
4. 에이전트를 디스패치한다.
5. 결과를 모아 단일 통합자가 충돌, 중복, 누락을 정리한다.
6. 수렴 후 VERIFY와 최종 리뷰를 다시 실행한다.

## 작업 계획 기록 형식

```md
## 병렬화 점검

- 모드: `SEQUENTIAL | PARALLEL_REVIEW | PARALLEL_IMPLEMENT | PARALLEL_INVESTIGATION`
- 결정: `parallel | sequential`
- 이유:
- 공유 계약 고정 여부: `yes | no | n/a`
- 겹치는 파일 여부: `yes | no`
- 트랜잭션 / 도메인 충돌 위험: `yes | no | n/a`
```

## 백엔드 병렬화 주의

아래 백엔드 작업은 가볍게 병렬화하지 않는다.

- 같은 애그리거트 내부 정책 변경
- 같은 애플리케이션 서비스/유스케이스의 트랜잭션 경계 변경
- 같은 계약의 리포지토리 쿼리, 잠금, 조회 전략 변경
- 마이그레이션/스키마/엔티티/리포지토리 변경이 하나의 단위인 경우
- 권한, 인증, 리소스 범위, 멱등성, 동시성 규칙이 얽힌 변경

백엔드 병렬화가 유용한 경우에도 보통 아래 순서를 따른다.

1. 스펙 또는 아키텍처 리뷰가 계약을 먼저 고정한다.
2. 테스트 리뷰가 RED 증거를 먼저 정의한다.
3. 파일과 책임이 분리될 때만 구현을 병렬화한다.
4. 구현 후 통합, 보안, 품질 리뷰는 병렬로 실행할 수 있다.
5. 최종 수렴은 한 통합자가 수행한다.

## 도메인 / 영속성 병렬 분리

`backend-domain-modeler`와 `backend-persistence-implementer`는 아래 조건을 만족할 때만 병렬로 실행한다.

- 애그리거트 경계와 리포지토리 계약이 활성 계획에 고정되어 있다.
- 두 에이전트가 같은 파일을 수정하지 않는다.
- 도메인 작업은 불변 조건과 상태 전이를 소유한다.
- 영속성 작업은 어댑터, 쿼리, 매핑, 잠금, 페이지네이션, 영속성 테스트를 소유한다.
- 트랜잭션 경계는 `backend-application-implementer` 또는 주 통합자가 수렴한다.
- 수렴 단계에서 VERIFY 전에 도메인/영속성 불일치를 확인한다.

아래 경우는 순차 실행한다.

- 영속성 모델과 도메인 모델이 같은 파일을 공유한다.
- 리포지토리 인터페이스가 바뀐다.
- 같은 애그리거트의 상태 전이와 쿼리 조건이 함께 바뀐다.
- 마이그레이션/스키마 변경과 도메인 불변 조건 변경이 강하게 연결되어 있다.
